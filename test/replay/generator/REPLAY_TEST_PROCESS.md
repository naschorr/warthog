# Replay Regression Test — Build Process

Step-by-step guide to create a new regression test file from a raw `.wrpl` replay, its in-game battle log, and the parsed replay JSON.

---

## Prerequisites / Inputs

| Artifact | Source | Description |
|---|---|---|
| **Raw replay** | War Thunder client `replays/` folder | A `.wrpl` binary file (e.g. `#2026.03.06 01.16.49.wrpl`) |
| **Battle log** | Transcribed from screenshots of the game client chat window | Plain-text file of all events (kills, deaths, awards, set-afire, etc.). **OCR/transcription may introduce small errors** — usernames, vehicle names, and timestamps should be cross-checked against the parsed JSON. |
| **Parsed replay JSON** | Generated by `warthog_replay_data_grabber.py --file <wrpl> --output <dir>` | The structured JSON output from the replay parser |
| **Vehicle data** | `data/vehicle_data/processed_vehicle_data/processed_vehicle_data.*.json` | Lookup table: `vehicle_id` → `{ "name": "Display Name", … }` |

---

## Trust Hierarchy (Critical)

This is the most important concept. Different sources have different trust levels:

| Data Point | Authoritative Source | Notes |
|---|---|---|
| Kill/death **totals** (`kills.ground`, `kills.air`, `deaths.total`) | **Parsed JSON** (from BLK results block) | These come from a well-understood part of the parser |
| **Team** assignment | **Parsed JSON** (from BLK) | |
| **Awards** list | **Parsed JSON** | Award IDs from the parsed replay are trusted |
| **Lineup** (vehicle IDs) | **Parsed JSON** | Lineup data comes from a well-understood part of the replay parser |
| Kill/death **details** (who killed whom, with what, when) | **Battle log ONLY** | The BL is the sole source of truth for event attribution. Do NOT use kill/death detail records from the parsed JSON — they may contain phantom entries, duplicates, or misattributions. |
| **Vehicle ID** for a display name in the BL | **Parsed JSON → player.lineup[]** | Cross-reference the BL display name (e.g. "Sherman VC") against the player's lineup to find the correct `vehicle_id` (e.g. `uk_sherman_vc_firefly`). Use `vehicle_data` JSON as a secondary lookup if needed (keyed by `vehicle_id`, has `"name"` field). |

### What counts as a kill/death in the Battle Log

Lines containing **`destroyed`** or **`shot down`** are kills — they generate both a `KillDetailTruth` for the killer and a `DeathDetailTruth` for the victim.

Lines containing **`has crashed`** or **`has been wrecked`** are **deaths with no killer**. They generate a `DeathDetailTruth` for the victim only (no kill detail for anyone), with `killer_username` and `killer_vehicle` left as `None`:

```python
# 6:50 =TDGL= SCHNOOKUMSPRIME (P-400) has crashed.
DeathDetailTruth("p-400", time_seconds=410),
```

These are **NOT** kills or deaths and must be ignored:
- `set afire`
- `severely damaged`
- `critically damaged`

## Step 0 — Set Up the Test Directory

```
test/replay/<session_id>/
├── <wrpl_filename>.wrpl              # Raw replay (copy from game)
├── battle_log_<session_id>.txt       # Battle log (transcribed from screenshots — may contain OCR errors)
├── replay_<date>_<time>_<session_id>.json  # Parsed JSON (generated)
├── notes.md                          # Optional context about the match
└── test_replay_<session_id>.py       # The test file (you create this)
```

1. Create the directory: `test/replay/<session_id>/`
2. Copy the `.wrpl` file into it.
3. Save the battle log as `battle_log_<session_id>.txt`.
4. Generate the parsed JSON:
   ```
   python src/replay_data_grabber/warthog_replay_data_grabber.py \
     --file "test/replay/<session_id>/<wrpl_filename>.wrpl" \
     --output "test/replay/<session_id>" \
     --overwrite
   ```

## Step 1 — Extract Metadata from Parsed JSON

Open the parsed JSON and record:

- `session_id` — e.g. `"63ef86a001440d4"`
- `start_time` — e.g. `"2026-03-06T00:52:48"` (UTC, used as battle start)
- `author.username` — the player who recorded the replay
- `len(players)` — player count (typically 32)

---

## Step 2 — Build the PLAYERS List: Scalars from Parsed JSON

For each player in `parsed_json["players"]`, extract:

```python
PlayerTruth(
    username=player["username"],
    team=player["team"],
    kills_ground=player["kills"]["ground"],
    kills_air=player["kills"]["air"],
    deaths_total=player["deaths"]["total"],
    awards=player["awards"],        # list of award ID strings
    kill_details=[...],             # populated in Step 4
    death_details=[...],            # populated in Step 4
    is_author=(player["username"] == author_username),
)
```

These scalar fields (`team`, `kills_ground`, `kills_air`, `deaths_total`, `awards`) come directly from the parsed JSON and are trusted.

---

## Step 3 — Build Lineup Lookup Maps

### Quick start: use the extraction script

A helper script dumps every player's lineup with resolved display names:

```bash
python test/replay/extract_lineups.py <parsed_replay.json>
```

Example:

```bash
python test/replay/extract_lineups.py test/replay/63ef86a001440d4/replay_2026-03-06_00-52-48_63ef86a001440d4.json
```

The script auto-discovers the latest vehicle data file in `data/vehicle_data/processed_vehicle_data/`. You can also pass `--vehicle-data <path>` explicitly. Output looks like:

```
# Session: 63ef86a001440d4
# Author:  Ranol
============================================================
  SCHNOOKUMSPRIME
  Team 2 | K(g)=2 K(a)=0 D=3
  Lineup (5 vehicles):
    - p-400  (P-400)
    - us_halftrack_m16  (M16 MGMC)
    - us_m24_chaffee  (M24)
    - us_m4a1_1942_sherman  (M4A1)
    - us_m10  (M10 GMC)
```

This gives you the `vehicle_id → display name` mapping you need for Step 4.

### Manual approach

For each player, their lineup is in `player["lineup"]`, a list of vehicle ID strings:

```python
# player["lineup"] = ["us_m4a1_1942_sherman", "us_m3_lee", "us_m3a1_stuart", "us_m4_sherman"]
```

To resolve a BL display name (e.g. `"Sherman VC"`) to a `vehicle_id`:
1. Look in the player's lineup list.
2. For each `vehicle_id` in the lineup, look up its display name in the vehicle data JSON (`data/vehicle_data/processed_vehicle_data/*.json`), which is keyed by `vehicle_id` and has a `"name"` field.
3. Match the BL display name against the `"name"` values (may need fuzzy matching for special characters like `⊙`, `☆`, `✸`, `★`, `🇨🇭`).

**Important**: BL display names often have cosmetic prefixes (e.g. `⊙T-34`, `☆M24`, `✸M24`, `★P-47D`, `🇨🇭Bf 109 F`) that denote that the player's country and the vehicle's country of origin aren't the same. For example the M24 is a US light tank, but when China uses it there's a star pattern in front of it (sometimes transcribed as ✸M24). Similarly when Italy uses it there's a roundel in front of it (sometimes transcribed as ⊙M24). Japan's M24 also can often have a star as well to represent their rising sun symbol (sometimes transcribed as ☆M24).

## Step 4 — Build Kill/Death Details from Battle Log

This is the most labor-intensive step. For every `destroyed` or `shot down` line in the BL:

### 4a. Parse each BL line

Format: `MM:SS <killer_display_name> (<killer_vehicle_display>) destroyed|shot down <victim_display_name> (<victim_vehicle_display>)`

Usernames in the BL have clan tags and emoji that must be stripped:
- `=TDGL= Pro_Gamer_20000` → `"Pro_Gamer_20000"`
- `🎮 channdro` → `"channdro"`
- `.Rdh1.Bymemory` → `"Bymemory"`
- `[Vaygr] Trember` → `"Trember"`
- `^UAOD^ ChasingRapture` → `"ChasingRapture"`
- `=Vizzy= 🎮 C DESTROYER26` → `"C DESTROYER26"`
- `=285AW= autumn1196` → `"autumn1196"`
- `[iCAT] ACCTGU 145` → `"ACCTGU 145"`
- `=JPs3V= _SP23` → `"_SP23"`
- `-4b0- Wiggle#6` → `"Wiggle#6"`
- `=IMPro= ooxxaa` → `"ooxxaa"`
- `=TROY= Лейтенант Ебонов` → `"Лейтенант Ебонов"`
- `=KPOHA= IBecameTheAmmo` → `"IBecameTheAmmo"`
- `=CNGDP= Late Noon` → `"Late Noon"`

**Clan tag patterns**: `=TAG=`, `[TAG]`, `^TAG^`, `-TAG-`, `.TAG.`, emoji prefixes. The stripped username must match the `username` field in the parsed JSON exactly.

### 4b. Convert BL timestamp to elapsed seconds

```
time_seconds = MM * 60 + SS
```

Example: `7:24` → `7 * 60 + 24 = 444`

The result is an **integer** (not a float).

### 4c. Resolve vehicle IDs

For each BL line, resolve both the killer and victim vehicle display names to `vehicle_id` strings using the lineup lookup from Step 3.

### 4d. Assign to the correct player

Each BL `destroyed`/`shot down` line generates **two** entries:
1. A **`KillDetailTruth`** for the **killer** player
2. A **`DeathDetailTruth`** for the **victim** player

```python
# BL line: "7:16 Ranol (SAV 20.12.48) destroyed 🎮 Darkcart 8430 (M4A2)"
# → Ranol gets a KillDetailTruth:
KillDetailTruth("sw_sav_fm48", "Darkcart 8430", "us_m4a2_sherman", 436)
# → Darkcart 8430 gets a DeathDetailTruth:
DeathDetailTruth("us_m4a2_sherman", "Ranol", "sw_sav_fm48", 436)
```

Each BL `has crashed`/`has been wrecked` line generates **one** entry:
- A **`DeathDetailTruth`** for the **victim** only, with no killer fields:

```python
# BL line: "6:50 =TDGL= SCHNOOKUMSPRIME (P-400) has crashed."
# → SCHNOOKUMSPRIME gets a DeathDetailTruth (no killer):
DeathDetailTruth("p-400", time_seconds=410)
```

### 4e. Add BL comment above each entry

Every `KillDetailTruth` and `DeathDetailTruth` must have the original BL line as a comment above it:

```python
kill_details=[
    # 7:16 Ranol (SAV 20.12.48) destroyed 🎮 Darkcart 8430 (M4A2)
    KillDetailTruth("sw_sav_fm48", "Darkcart 8430", "us_m4a2_sherman", 436),
],
```

### 4f. Sort within each player

Kill details and death details should be sorted by `time_seconds` (chronological order).

## Step 5 — Write the Test File

### File structure

```python
"""
Regression test for parsed replay session <session_id>.
...docstring...
"""

from __future__ import annotations
import json
from collections import Counter
from datetime import datetime, timezone
from pathlib import Path
from typing import Any
import pytest
from src.common.factories import ServiceFactory
from common.replay_test_helpers import (
    DEFAULT_TIMESTAMP_TOLERANCE_S,
    DeathDetailTruth,
    KillDetailTruth,
    PlayerTruth,
    elapsed_seconds,
    find_death,
    find_kill,
)

PLAYERS: list[PlayerTruth] = [
    # ... all PlayerTruth entries from Steps 2+4 ...
]

WRPL_PATH = Path(__file__).parent / "<wrpl_filename>.wrpl"

@pytest.fixture(scope="module")
def replay() -> dict[str, Any]:
    factory = ServiceFactory()
    parser = factory.get_replay_parser_service()
    parsed = parser.parse_replay_file(WRPL_PATH)
    return json.loads(parsed.model_dump_json())

@pytest.fixture(scope="module")
def players_by_name(replay: dict[str, Any]) -> dict[str, dict[str, Any]]:
    return {p["username"]: p for p in replay["players"]}

@pytest.fixture(scope="module")
def battle_start(replay: dict[str, Any]) -> datetime:
    dt = datetime.fromisoformat(replay["start_time"])
    if dt.tzinfo is None:
        dt = dt.replace(tzinfo=timezone.utc)
    return dt

class TestMetadata:
    def test_session_id(self, replay):
        assert replay["session_id"] == "<session_id>"
    def test_player_count(self, replay):
        assert len(replay["players"]) == <N>
    def test_author_username(self, replay):
        author_truth = next(p for p in PLAYERS if p.is_author)
        assert replay["author"]["username"] == author_truth.username

@pytest.mark.parametrize("truth", PLAYERS, ids=lambda p: p.username)
class TestAllPlayers:
    # ... test methods (see template below) ...
```

### TestAllPlayers test methods

| Test | What it asserts |
|---|---|
| `test_team` | `player["team"] == truth.team` |
| `test_kills_ground` | `player["kills"]["ground"] == truth.kills_ground` |
| `test_kills_air` | `player["kills"]["air"] == truth.kills_air` |
| `test_deaths_total` | `player["deaths"]["total"] == truth.deaths_total` |
| `test_awards_present` | Every award ID in `truth.awards` appears in parsed awards (skips if empty) |
| `test_award_counts` | Each award appears at least N times where N = count in `truth.awards` |
| `test_kill_details` | Each `KillDetailTruth` can be matched in `player["kills"]["vehicles"]` |
| `test_death_details` | Each `DeathDetailTruth` can be matched in `player["deaths"]["vehicles"]` |

### Author vs non-author assertion behavior

- **Author** (`is_author=True`): Kill/death detail mismatches are **hard assertion failures**.
- **Non-author** (`is_author=False`): Kill/death detail mismatches are **`pytest.xfail(strict=False)`** — reported but don't fail the suite.

---

## Step 6 — Verify and Iterate

1. Run the tests: `pytest test/replay/<session_id>/ -v`
2. Check for:
   - **Syntax errors** — fix immediately
   - **Scalar failures** (team, kills, deaths) — these should pass from Step 2 data; if not, investigate
   - **Award failures** — compare parsed JSON awards against the test; these should pass directly
   - **Kill/death detail xfails** — expected for non-author players where the parser may not resolve all events
   - **Author kill/death failures** — investigate; the BL-derived truths should match parsed output for the author
3. Iterate on any issues.

## Shared Helpers Reference

The shared module `test/replay/common/replay_test_helpers.py` provides:

### Dataclasses

- **`KillDetailTruth(killer_vehicle, victim_username, victim_vehicle, time_seconds)`**
  - `killer_vehicle`: vehicle_id of the killer (required)
  - `victim_username`: stripped username of the victim (optional)
  - `victim_vehicle`: vehicle_id of the victim (optional)
  - `time_seconds`: elapsed seconds from battle start as `int` (optional)

- **`DeathDetailTruth(victim_vehicle, killer_username, killer_vehicle, time_seconds)`**
  - `victim_vehicle`: vehicle_id of the victim (required)
  - `killer_username`: stripped username of the killer (optional)
  - `killer_vehicle`: vehicle_id of the killer (optional)
  - `time_seconds`: elapsed seconds from battle start as `int` (optional)

- **`PlayerTruth(username, team, kills_ground, kills_air, deaths_total, awards, kill_details, death_details, is_author)`**

### Functions

- **`find_kill(kills, truth, battle_start, tolerance)`** — Matches a `KillDetailTruth` against parsed kill vehicle list. Tolerance default: ±5 seconds.
- **`find_death(deaths, truth, battle_start, tolerance)`** — Same for deaths.
- **`elapsed_seconds(time_utc_str, battle_start)`** — Converts a `time_utc` ISO string from parsed JSON to elapsed seconds.

## Parsed JSON Structure Reference

```
replay (top-level):
  session_id: str
  start_time: str (ISO 8601, UTC)
  author: { username: str, ... }
  players: [
    {
      username: str
      team: int (1 or 2)
      lineup: [str, ...]                    # list of vehicle_id strings
      kills: {
        air: int
        ground: int
        vehicles: [                         # ⚠️ DO NOT TRUST for ground truth
          {
            killer_vehicle: str
            victim_username: str
            victim_vehicle: str
            time_utc: str (ISO 8601)
          }, ...
        ]
      }
      deaths: {
        total: int
        vehicles: [                         # ⚠️ DO NOT TRUST for ground truth
          {
            victim_vehicle: str
            killer_username: str
            killer_vehicle: str
            time_utc: str (ISO 8601)
          }, ...
        ]
      }
      awards: [str, ...]                    # list of award ID strings
    }, ...
  ]
```

**⚠️ WARNING**: `kills.vehicles` and `deaths.vehicles` from the parsed JSON are **NOT trusted** as ground truth. They may contain:
- Phantom kills (events that never happened in the BL)
- Duplicate entries
- Misattributed events (wrong killer/victim)
- Missing events (BL events not in the parsed data)

Only use them to verify that the parser CAN find them — the BL is what defines what SHOULD be there.

## Common Pitfalls

1. **Don't use parsed JSON kill/death details as truth** — Only the battle log defines what kills/deaths actually happened.
2. **Use `int` for `time_seconds`** — Not floats. BL timestamps are `MM:SS`, so `MM*60 + SS` is always an integer.
3. **Strip clan tags from BL usernames** — The username in the test must match the parsed JSON's `username` field exactly.
4. **Some BL display names have cosmetic prefixes** — `⊙`, `☆`, `✸`, `★`, `🇨🇭f` etc. are NOT part of the vehicle name for lookup purposes.
5. **Each BL kill line generates TWO test entries** — One kill detail for the killer, one death detail for the victim.
6. **Sort entries chronologically** within each player's `kill_details` and `death_details`.
7. **`destroyed` and `shot down` = kill + death; `crashed` and `wrecked` = death only** — `set afire`, `severely damaged`, `critically damaged` are NOT kills/deaths. Crashed/wrecked entries generate a `DeathDetailTruth` with no killer fields (only `victim_vehicle` and `time_seconds`).
8. **Non-ASCII usernames** — Use Unicode escapes in Python string literals if needed (e.g. `"\u041b\u0435\u0439\u0442\u0435\u043d\u0430\u043d\u0442 \u0415\u0431\u043e\u043d\u043e\u0432"` for `Лейтенант Ебонов`).
9. **Battle log transcription errors** — BL text files are generated by transcribing screenshots (OCR), so they may contain small errors in usernames, vehicle names, or timestamps. Always cross-check BL names against the parsed JSON `username` fields and vehicle data display names. Common OCR issues: `l`/`1`/`I` confusion, missing special characters, garbled Unicode, extra/missing spaces.

## Example Workflow Summary

```
1. Copy .wrpl into test/replay/<session_id>/
2. Capture battle log → battle_log_<session_id>.txt
3. Run replay parser → generates parsed JSON
4. From parsed JSON: extract session_id, start_time, author, player scalars, lineups, awards
5. From battle log: extract every "destroyed"/"shot down"/"has crashed"/"has been wrecked" line
6. For each BL kill line:
   a. Strip clan tags from usernames
   b. Convert MM:SS → integer seconds
   c. Resolve vehicle display names → vehicle_ids via lineup + vehicle_data
   d. Create KillDetailTruth for killer, DeathDetailTruth for victim
   e. Add BL line as comment above each entry
7. Assemble PLAYERS list with scalars + kill/death details
8. Write test file using template (fixtures, TestMetadata, TestAllPlayers)
9. Run pytest, verify, iterate
```
